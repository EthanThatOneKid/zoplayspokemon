# Public Positioning

This project should be framed as an educational case study about a real collaborative Pokemon emulator interface on hosted infrastructure. The public value is in documenting what was built, what it demonstrates technically, and why repeating it is not being encouraged here.

## What the public story should emphasize

- this is a record, not a recipe
- coordinated remote input
- room-based multiplayer control patterns
- frame refresh strategy
- long-poll state updates
- latency handling and presentation timing
- emulator service orchestration on Zo
- the legal/product tension is part of the case study, not something to hide

## What public-facing materials should avoid

- presenting the repo as a tutorial, starter kit, or endorsement of imitation
- encouraging people to recreate or deploy the same Pokemon-based experience
- treating non-commercial use as if it removes copyright risk
- implying that transparency about the build is the same as permission to repeat it

## Explicit boundary language

- say plainly that the repo is proof of what is possible, not a guide for doing it again
- say plainly that repeating the process with copyrighted games may create legal risk
- keep Pokemon named when it is necessary for honest documentation of the case study
- distinguish between describing the deployed system and instructing readers how to reproduce it

## Open follow-ups

- audit route copy, screenshots, and repo metadata so they match the "record, not recipe" stance
- review technical sections for accidental step-by-step reproduction guidance
- keep the README intent statement and legal-risk disclaimer near the top as the canonical framing
