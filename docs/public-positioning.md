# Public Positioning

This project is being reframed as an educational case study about collaborative emulator UX on hosted infrastructure.

## What the public story should emphasize

- coordinated remote input
- room-based multiplayer control patterns
- frame refresh strategy
- long-poll state updates
- latency handling and presentation timing
- emulator service orchestration on Zo

## What public-facing materials should avoid

- presenting the project as a way to play Pokemon online together
- leading with Nintendo branding, screenshots, or game-specific copy
- encouraging public redistribution of commercial game content
- treating non-commercial use as if it removes copyright risk

## Safer defaults

- use `roms/PlantBoy.gb` or other clearly redistributable demo content in public examples
- describe the system as a technical experiment or architecture demo
- keep game-specific/private experimentation separate from the public project narrative
- review repo metadata, screenshots, and live route copy before sharing links widely

## Open follow-ups

- decide whether the live route should stay public as-is, move to a safer demo ROM, or become private
- audit route copy and visible assets for game-specific branding
- separate generic emulator infrastructure from game-specific experiments over time
